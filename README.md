# vaka-registry

The official [vaka](https://github.com/infrasecture/vaka) recipes registry.

A **recipe** is a ready-to-run, security-hardened compose project: a
`compose.yaml` plus a `vaka.yaml` egress policy, together with any supporting
files. Every top-level directory in this repository is one recipe.

## Using recipes

Once `vaka get` ships (see the
[design](https://github.com/infrasecture/vaka/blob/main/docs/design/recipes-registry.md)):

```bash
vaka search agent          # browse the catalog
vaka get codex             # fetch the recipe into ./codex, verified
cd codex && vaka up        # vaka never runs a recipe by itself
```

Until then, recipes can be used by cloning this repository and running `vaka
up` inside a recipe directory.

## Authoring

Each recipe directory must contain:

- `recipe.yaml` — the manifest (`apiVersion: recipes.vaka/v1alpha1`,
  `kind: Recipe`); `name` must equal the directory name (`[a-z0-9-]+`) and
  `version` is strict per-recipe SemVer, bumped on every content change.
- `compose.yaml` (and/or `docker-compose.yaml`) — the compose project.
- `vaka.yaml` — the egress policy for every service.
- `README.md` — what the recipe runs and why its policy looks the way it does.

Conventions:

- Never commit secrets or `.env`; ship `*.example` templates for anything a
  user is expected to edit. Users keep customizations in untracked files
  (`.env`, `compose.override.yaml`) so `vaka get` updates never conflict.
- The `.vaka-*` name prefix is reserved for vaka's own state; recipes must
  not ship such paths.
- The `provides:` and `requires:` manifest fields are reserved for future
  recipe composability and are rejected until specified.

Distribution artifacts (`index.yaml` catalog + per-recipe release tarballs
with sha256 digests) are generated by CI; see the design document for the
full registry format and security model.
